## Model

Pirate has three classes of Turso credentials:

1. platform control-plane credential
2. central control-plane database runtime credential
3. high-cardinality per-community database runtime credentials

Only the first two belong in Infisical as root secrets.

Per-community credentials are too numerous to manage as independent Infisical entries. They must be generated by the control plane and stored encrypted in the central control-plane database.

## Root Secrets

### dev:/services/control-plane

Secrets used by a private control-plane worker or human-run provisioning tooling.

| Name | Type | Purpose | Consumer | Rotation |
|---|---|---|---|---|
| `TURSO_PLATFORM_API_TOKEN` | `api-key` | Create groups, create databases, mint auth tokens, transfer groups, rotate credentials | private control-plane worker or human-run tooling only | on compromise and after initial setup handoff |

Rules:

- this secret must not be present in the public API worker runtime
- this secret is the root capability for provisioning and transfer
- do not paste this secret into chat or commit it anywhere

### dev:/services/api

Secrets used by the public API worker runtime.

| Name | Type | Purpose | Consumer | Rotation |
|---|---|---|---|---|
| `TURSO_CONTROL_PLANE_AUTH_TOKEN` | `api-key` | Connect the API worker to the central Pirate control-plane database | API worker | on compromise or planned rotation |
| `TURSO_COMMUNITY_DB_WRAP_KEY` | `worker-secret` | Encrypt and decrypt stored per-community database credentials in the central DB | API worker | on compromise, with controlled rewrap |

Rules:

- `TURSO_COMMUNITY_DB_WRAP_KEY` is not a Turso token
- it is an application-managed envelope-encryption key
- the API worker may read and use encrypted community DB credentials only after decrypting them in memory

## Non-Secret Config

These values should be version-controlled or ordinary worker env config, not Infisical secrets:

- `TURSO_ORGANIZATION_SLUG`
- `TURSO_CONTROL_PLANE_DATABASE_URL`
- `TURSO_CONTROL_PLANE_DATABASE_NAME`
- community group naming convention
- community primary database naming convention

Reasoning:

- database URLs/hostnames are not secrets when auth tokens are separate
- stable naming and routing config should stay auditable in version control

## High-Cardinality Community Credentials

Per-community database credentials are special.

They should not be modeled as:

- one Infisical secret per community
- one environment variable per community

Instead:

1. The control plane creates a community group.
2. The control plane creates the primary database in that group.
3. The control plane mints a database auth token for the primary database.
4. The control plane encrypts that token using `TURSO_COMMUNITY_DB_WRAP_KEY`.
5. The encrypted token, database URL, key version, and rotation metadata are stored in the central control-plane database.

Recommended v0 posture:

- runtime uses database-scoped tokens for the primary community database
- group tokens are reserved for migration, branching, or operator tooling where broader scope is intentional

Relevant Turso docs:

- create database token: <https://docs.turso.tech/api-reference/databases/create-token>
- create group token: <https://docs.turso.tech/api-reference/groups/create-token>
- invalidate all DB tokens: <https://docs.turso.tech/api-reference/databases/invalidate-tokens>
- invalidate all group tokens: <https://docs.turso.tech/api-reference/groups/invalidate-tokens>

The durable storage contract for these credentials lives in [turso-control-plane-schema.md](/home/t42/Documents/pirate-v2/docs/turso-control-plane-schema.md).

## Rotation

### Platform token

- rotate in Infisical
- redeploy or restart the control-plane consumer
- verify new provisioning still works
- invalidate old token if appropriate

### Central control-plane DB token

- mint a new DB token
- update Infisical
- redeploy the API worker
- invalidate old DB tokens

### Community DB token

- mint a new DB token for that community
- encrypt with the current wrap key
- update the encrypted record in the central DB
- cut over the worker to the new credential
- invalidate old DB tokens for that community

### Wrap key

Wrap-key rotation is a re-encryption event:

1. load each encrypted community credential
2. decrypt with the old wrap key
3. re-encrypt with the new wrap key
4. store the new ciphertext and key version
5. only then retire the old wrap key

This must be handled as an explicit migration, not an ad hoc env change.

## Community Transfer

When a community group is transferred to another Turso organization:

1. freeze structural mutations for that community
2. mint fresh credentials under the receiving organization
3. update Pirate routing and encrypted connection metadata
4. invalidate old group or database tokens when the cutover is complete

Turso documents that existing URLs and tokens may continue to work after transfer, but Pirate should treat those as temporary migration artifacts and rotate to new credentials promptly.

Relevant Turso docs:

- transfer group: <https://docs.turso.tech/api-reference/groups/transfer>

## Local Development

Recommended local dev stack:

- use a local SQLite/libSQL file for routine development
- use `turso dev` when libSQL server behavior matters
- avoid using hosted Turso dev databases for routine local iteration

Relevant docs:

- <https://docs.turso.tech/local-development>
